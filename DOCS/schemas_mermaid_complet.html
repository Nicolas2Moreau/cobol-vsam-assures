Architecture V1 — VSAM Pur

Version initiale du projet avec appel statique à PGMVSAM. Le programme principal MAJASSU appelle l'accesseur VSAM de manière hardcodée. Tous les fichiers (assurés et mouvements) sont gérés via VSAM uniquement.

Architecture V2 — Accesseur Dynamique

Version évoluée avec choix de l'accesseur à l'exécution via PARM JCL. MAJASSU V2 peut appeler soit PGMVSAM (VSAM) soit PGMDB2 (DB2) sans recompilation. Le programme KSTODB2 assure la synchronisation entre le KSDS et la table DB2.

    B -->|"'PGMVSAM'"| C[PGMVSAM
Accesseur VSAM]
    B -->|"'PGMDB2'"| D[PGMDB2
Accesseur DB2]
    C -->|"CRUD VSAM"| E[(KSDS ASSURES3)]
    D -->|"CRUD SQL"| F[(Table DB2
API12.ASSURES)]
    G[KSTODB2
Synchronisation] -->|"READ KSDS"| E
    G -->|"TRUNCATE + INSERT"| F
    A -->|"Anomalies"| H[GDG ETATANO]

Les Mappers — Cœur de l'Abstraction

Le rôle crucial des mappers : traduire la complexité technique (File Status VSAM et SQLCODE DB2) en 6 codes retour simples. PGMVSAM utilise un seul mapper universel, tandis que PGMDB2 nécessite 4 mappers contextuels car le même SQLCODE peut avoir des significations différentes selon l'opération. 

graph TB
    subgraph VSAM["PGMVSAM - 1 Mapper Unique"]
        V1[File Status VSAM] --> V2{MAPPER-FILE-STATUS}
        V2 -->|"'00'"| V3["00 OK"]
        V2 -->|"'10'"| V4["04 EOF"]
        V2 -->|"'22'"| V5["02 DUPLICATE"]
        V2 -->|"'23'"| V6["01 NOTFOUND"]
        V2 -->|"'9x', '35'"| V7["03 IOERROR"]
        V2 -->|"autres"| V8["99 ERROR"]
    end
    
    subgraph DB2["PGMDB2 - 4 Mappers Contextuels"]
        D1[SQLCODE DB2] --> D2{Quelle fonction?}
        D2 -->|"f03, f05
READ/DELETE"| D3[MAPPER-READ]
        D2 -->|"f04, f06, f09
UPDATE/INSERT/TRUNCATE"| D4[MAPPER-WRITE]
        D2 -->|"f08
FETCH"| D5[MAPPER-FETCH]
        D2 -->|"f07
OPEN CURSOR"| D6[MAPPER-OPEN]
        
        D3 -->|"0 → 00
+100 → 01
-501 → 03"| D7[Codes retour]
        D4 -->|"0 → 00
-803/-811 → 02
-501 → 03"| D7
        D5 -->|"0 → 00
+100 → 04
-501 → 03"| D7
        D6 -->|"0 → 00
autres → 99"| D7
    end
    
    V3 --> RET[LS-CODE-RETOUR]
    V4 --> RET
    V5 --> RET
    V6 --> RET
    V7 --> RET
    V8 --> RET
    D7 --> RET
    
    style V2 fill:#F39C12
    style D2 fill:#E74C3C,color:#fff
    style RET fill:#27AE60,color:#fff

Workflow JCL Global

Vue d'ensemble du workflow opérationnel complet. Phase 1 : Initialisation unique (GDG, VSAM, DB2). Phase 2 : Exécution courante avec choix V1 ou V2. Phase 3 : Resynchronisation DB2 si nécessaire. 

graph TB
    subgraph PHASE1["PHASE 1 - INITIALISATION (Une seule fois)"]
        A[JCREGDG] -->|"Création GDG"| B[JCREVSAM]
        B -->|"Définition + Chargement
KSDS + ESDS"| C[SPUFI]
        C -->|"CREATE TABLE
API12.ASSURES"| D[DCLGEN]
        D -->|"Génération
DCLASSU.cpy"| E[JCOMPDB2]
        E -->|"Précompilation DB2
Compilation
BIND"| F[JKSDB2]
        F -->|"Chargement initial
KSDS → DB2"| INIT_END[Initialisation terminée]
    end
    
    subgraph PHASE2["PHASE 2 - EXÉCUTION COURANTE"]
        G{Version?} -->|"V1"| H[JMAJMVT]
        G -->|"V2"| I{Accesseur?}
        I -->|"PARM='PGMVSAM'"| J[JMAJMV2
avec PGMVSAM]
        I -->|"PARM='PGMDB2'"| K[JMAJDB2
avec PGMDB2]
        
        H --> RUN_END[Exécution terminée]

    end

GitFlow — Stratégie de Branches

Workflow Git du projet avec nomenclature Angular. Branche main (production), test (validation CI/CD), gnucobol (portabilité), et feature branches pour le développement. Les commits suivent la convention feat/fix/docs/refactor/test/chore. 

%%{init: { 'theme': 'base', 'gitGraph': {'showBranches': true, 'showCommitLabel':true}}}%%
gitGraph
    commit id: "Initial commit"
    commit id: "feat: V1 PGMVSAM"
    commit id: "feat: V1 MAJASSU"
    
    branch test
    checkout test
    commit id: "feat: add tests"
    
    checkout main
    merge test tag: "v1.0"
    
    branch gnucobol
    checkout gnucobol
    commit id: "feat: GnuCOBOL compat"
    commit id: "fix: Windows paths"
    
    checkout main
    commit id: "feat: V2 PGMDB2"
    
    branch feat-db2
    checkout feat-db2
    commit id: "docs: DB2.md"
    commit id: "feat: KSTODB2"
    
    checkout test
    merge feat-db2
    commit id: "test: CI syntax check"
    
    checkout main
    merge test tag: "v2.0"
    
    checkout test
    commit id: "refactor: code cleanup"
    
    checkout main
    commit id: "chore: workflows"
    commit id: "docs: RECAPV2.md" tag: "2026-02-24"

🎨 Nomenclature Angular

feat: Nouvelle fonctionnalité
fix: Correction de bug
docs: Documentation
refactor: Restructuration code
test: Tests
chore: Maintenance
